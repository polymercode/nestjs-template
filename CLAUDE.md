# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm run start:dev          # dev server, watch mode
npm run build               # compile to dist/ (uses tsconfig.build.json)
npm test                    # unit tests (Jest, co-located *.spec.ts under src/)
npm run test:watch
npm run test:cov
npm run test:e2e            # e2e tests (test/*.e2e-spec.ts) — no real DB/Redis needed
npm run lint                 # ESLint with --fix

npm run db:migrate          # create/apply a dev migration
npm run db:deploy           # apply migrations (production, no prompts)
npm run db:seed             # seed admin@example.com / Admin123!
npm run db:studio
npm run db:generate         # regenerate Prisma client into src/generated/prisma
```

Run a single test file: `npx jest path/to/file.spec.ts` (unit) or
`npx jest --config ./test/jest-e2e.json path/to/file.e2e-spec.ts` (e2e).

Local infra: `docker compose up -d db redis` (Postgres + Redis only, no app container) is
the usual dev loop — then run the app with `npm run start:dev` outside Docker.

Before ending a session, stop anything you started: kill local `node`/`nest start`
processes bound to port 3000, and `docker compose down` any containers you brought up.

## Architecture

Layout: `src/main.ts` bootstraps `AppModule`; `src/app.setup.ts` holds the global pipe/CORS/
helmet/versioning config shared between `main.ts` and the e2e tests (`configureApp()` is
called from both, so changes to request-pipeline behavior belong there, not in `main.ts`).

**Global cross-cutting providers are registered once in `app.module.ts`** via `APP_GUARD` /
`APP_FILTER` / `APP_INTERCEPTOR`, in this order: `ThrottlerGuard` → `JwtAuthGuard` →
`RolesGuard`. Because of that order, a request is rate-limited before it's even checked for
a valid JWT.

- **Auth is opt-out, not opt-in.** `JwtAuthGuard` is global, so every route requires a valid
  access token unless the handler or controller has `@Public()`
  (`src/common/decorators/public.decorator.ts`). When adding a new public route, add
  `@Public()` explicitly — forgetting it is the default-safe direction, but forgetting to
  add it when you *meant* the route to be public is the common mistake.
- **RBAC is additive.** `@Roles(Role.ADMIN)` (`src/common/decorators/roles.decorator.ts`)
  only restricts a route if present; routes without it are unaffected by `RolesGuard`.
- **Rate limiting has two profiles**, configured in `src/throttler/throttler.module.ts`:
  a generous `default` profile applied globally, and a stricter `auth` profile applied to
  `AuthController` via `@Throttle({ auth: {} })` (register/login/refresh/logout are
  brute-force targets). Add new brute-force-sensitive endpoints to the `auth` profile the
  same way.
- **CORS** origins come from `CORS_ORIGIN` (comma-separated) via `app.corsOrigins` config;
  a bare `*` is allowed but logs a warning when `NODE_ENV=production`.

### Prisma 7 specifics

The generated client lives in `src/generated/prisma` (gitignored, regenerated by
`postinstall` / `npm run db:generate`), **not** `node_modules/@prisma/client` — imports are
`from '../../generated/prisma/client'` / `.../enums`. The connection is made via
`@prisma/adapter-pg` (`PrismaService` in `src/prisma/prisma.service.ts`), not a bundled
query engine binary.

Two Prisma 7 non-obvious requirements baked into this setup, don't remove them:

- `prisma/schema.prisma`'s generator block has `moduleFormat = "cjs"`. Without it, Prisma 7
  emits `import.meta.url` in the generated client even when compiled to CommonJS, which
  crashes at runtime (`exports is not defined in ES module scope`). The datasource URL is
  *not* in `schema.prisma` — it's resolved via `prisma.config.ts` (`env('DATABASE_URL')`),
  which is the Prisma 7 config model.
- `prisma/seed.ts` runs via `tsx`, not `ts-node`. `ts-node`'s CommonJS require hook can't
  resolve the `.js`-suffixed relative imports Prisma's generated client uses (they only
  resolve once `tsc` has actually compiled `.ts`→`.js`); `tsx` handles this natively.
- Jest has the same `.js`-extension problem for anything that imports the generated client
  (i.e. anything importing `PrismaService`). Both `package.json`'s `jest.moduleNameMapper`
  and `test/jest-e2e.json` carry `"^(\\.{1,2}/.*)\\.js$": "$1"` to strip the extension so
  Jest's resolver can find the `.ts` source. If a new test suite mysteriously fails with
  `Cannot find module './internal/class.js'`, it's missing this mapping.
- `tsconfig.build.json` has `incremental: false`. Leaving TS incremental builds on for the
  Nest build caused a real bug: an interrupted build leaves a stale `.tsbuildinfo` that
  makes the next `tsc` believe `dist/` is already up to date and skip re-emitting files
  entirely (silently producing a `dist/` with only `.d.ts` files, no `.js`). This bit both
  local `npm run build` and the Docker build.

### Testing overrides for global infra modules

`CacheModule` and `ThrottlerModule` both construct real clients (Redis) inside a
`registerAsync`/`forRootAsync` factory. Overriding just the token
(`.overrideProvider(CACHE_MANAGER)`) is **not enough** in e2e tests — the factory still
runs during module compilation and throws trying to reach Redis. Use `.overrideModule(...)`
to replace the whole module instead (see `test/app.e2e-spec.ts`'s `TestCacheModule`).
`PrismaService` is swapped the same way but via `overrideProvider`, since it's a plain
injectable, not a dynamic module — an in-memory `Map`-backed mock stands in for the
database so e2e tests don't need Postgres or Redis running.

### Docker / environments

Three compose files, not one with profiles: `docker-compose.yml` (local dev, inline env
vars, ports published for `db`/`redis` so you can connect a GUI client) vs
`docker-compose.production.yml` / `docker-compose.staging.yml` (secrets via
`env_file: .env.production` / `.env.staging` — copy from the `.example` versions — `db`
and `redis` have **no** published ports, reachable only over the compose network by `app`).
`NODE_ENV=staging` is a distinct valid value (see `env.validation.ts`) specifically so
Swagger stays enabled in staging while it's disabled in production (`main.ts` checks
`app.env !== 'production'`).

The `Dockerfile` is a 3-stage build (`build` / `prod-deps` / `runtime`). Both `build` and
`prod-deps` set a placeholder `DATABASE_URL` — `prisma generate` validates that the env var
resolves even though generation never opens a connection. `prod-deps` deliberately does
*not* pass `--ignore-scripts`, so Prisma's engine binaries download at build time (as root)
instead of lazily at container startup, which would otherwise fail under the non-root
runtime user.
